Store Partnership — Flow & API Reference

Dokumentasi lengkap untuk semua flow bisnis dan API yang sudah dibangun: transaksi POS per store, piutang fee bulanan (royalty/profit share), order material antar store, ongkir terpisah, dan dua endpoint integrasi eksternal.

Bagaimana semua terhubung

Satu Store, tiga jalur invoice

Semua flow berawal dari entitas Store. Dari satu store, ada tiga jalur transaksi yang masing-masing bermuara ke Sales Invoice terpisah — sengaja tidak pernah digabung, supaya piutang penjualan, fee kemitraan, dan ongkir masing-masing bisa ditagih dan direkonsiliasi sendiri-sendiri.

Batasan yang perlu diketahui

fee_percentage di Store Type adalah satu angka untuk semua store bertipe sama. Kalau tarif riil berbeda per store (biasanya tercatat di PKS Agreement.royalty_percent / profit_share_percent), Store Fee Statement belum otomatis membaca angka per-PKS itu. Detail di § Batasan & Ide Lanjutan.

Jalur 1

Transaksi POS: field store & filter by Company/Store

Masalah awal: transaksi POS bisa dilihat di invoice tapi tidak bisa difilter per company/store.

Solusi: custom field store (Link → Store) ditambahkan ke POS Invoice (kalau site pakai doctype terpisah) dan Sales Invoice (kalau site pakai mode "Sales Invoice untuk POS", is_pos=1) — keduanya in_list_view dan in_standard_filter, otomatis jadi kolom & filter standar sejajar dengan Company.

Hook validate (pos_invoice_store.set_store_from_warehouse) mengisi store otomatis dari kombinasi set_warehouse + company, dicocokkan ke Store.warehouse. Tidak menimpa kalau sudah diisi manual.

Konfirmasi lingkungan

Site ini terkonfirmasi pakai mode "Sales Invoice untuk POS", bukan doctype POS Invoice terpisah. Semua perhitungan total penjualan di dokumen ini (§ Piutang Fee Bulanan, § API) menghitung dari kedua sumber sekaligus supaya tetap kompatibel kalau mode site berubah.

Jalur 1 — Muara

Piutang fee bulanan (Store Fee Statement)

Menghitung & menagihkan royalty/profit-share bulanan dari store partner ke company, otomatis jadi piutang riil di AR Aging.

Store yang eligible

Store dengan partner terisi dan Store Type.settlement_model ≠ None. Store tipe OWN (settlement None) otomatis dilewati.

Alur generate

  1. Cek duplikat.

    Skip kalau sudah ada Store Fee Statement untuk store + periode yang sama — proses ini idempotent, aman dipanggil berulang.

  2. Hitung (calculate_statement).

    total_sales = SUM(grand_total) dari POS Invoice + Sales Invoice (is_pos=1), submitted & bukan return, untuk store & periode itu. fee_amount = total_sales × fee_percentage / 100. Status → ● Calculated.

  3. Tagih (kalau fee_amount > 0).

    Buat Sales Invoice baru ke Store.partner, 1 baris item (fee_item), langsung insert + submit. Status → ● Invoiced, field sales_invoice terisi.

Cara trigger

CaraKapan jalan
Tombol "Generate Now" di Store Partnership SettingsKapan saja, manual, periode bebas (default bulan lalu)
Tombol "Calculate" → "Create Sales Invoice" di form Store Fee StatementPer-store, satu-satu, untuk koreksi/kasus khusus
Scheduler bulanan (tasks.monthly)Hanya kalau store_fee_generation = "Cron (Automatic Monthly)". Default Manual → scheduler tidak melakukan apa-apa.

Status

● Draft baru dibuat → ● Calculated sudah dihitung, belum ditagih → ● Invoiced Sales Invoice sudah submit, tidak bisa diubah lagi.

Jalur 2

Order material dari store

Custom field store di Sales Order (in_list_view, in_standard_filter). Hook validate (sales_order_store.apply_store_defaults) — begitu store dipilih, otomatis isi field yang masih kosong saja (tidak menimpa input manual):

  • customer ← Store.partner
  • company ← Store.company
  • selling_price_list ← Store Type.default_price_list
  • taxes_and_charges ← resolve dari Store Type.default_tax_rule → Tax Rule.sales_tax_template

Setelah itu alur normal ERPNext: Sales Order → Delivery Note → Sales Invoice (piutang material, terpisah dari fee & ongkir).

Internal · dipakai oleh API §7.2

Manajemen sesi POS

ERPNext mewajibkan setiap Sales Invoice dengan is_pos=1 terkait ke POS Opening Entry yang berstatus Open dan tanggalnya hari ini — hanya boleh ada satu per POS Profile. Modul pos_session.get_or_open_pos_opening_entry() menangani ini otomatis, caller (API §7.2) tidak perlu tahu mekanismenya.

  1. Cari sesi yang open untuk POS Profile tsb.
  2. Kalau ada & tanggalnya hari ini → dipakai langsung.
  3. Kalau ada tapi tanggalnya kemarin (sesi lupa ditutup) → otomatis ditutup dulu pakai make_closing_entry_from_opening() bawaan ERPNext (hitung ulang dari invoice-invoice yang masuk di sesi itu), baru buka sesi baru untuk hari ini.
  4. Kalau ada lebih dari satu yang open (kondisi tak terduga) → berhenti dengan error jelas, minta admin selesaikan manual di Desk. Sengaja tidak auto-close banyak sekaligus tanpa pengawasan.
  5. Kalau tidak ada sama sekali → buka baru, opening_amount=0 untuk tiap payment method di POS Profile itu.
Integrasi eksternal

API Reference

Kedua endpoint di bawah adalah method whitelisted Frappe, dipanggil lewat REST:

POST /api/method/store_partnership.api.<nama_fungsi>
Authorization: token <api_key>:<api_secret>
Content-Type: application/json

API key/secret dibuat per User lewat Desk (User → API Access → Generate Keys). Permission mengikuti role user tsb.

Penting — idempotency

Selalu kirim pos_reference unik per transaksi dari sisi POS store. Tanpa ini, retry otomatis dari sistem POS (krn timeout dsb) akan membuat Sales Invoice dobel untuk penjualan yang sama.

Untuk diskusi lanjutan

Batasan & ide lanjutan

!

Fee percentage per-store vs per-Store-Type. Kalau tarif riil beda per store, Store Fee Statement perlu diarahkan baca dari PKS Agreement aktif milik store itu, bukan dari Store Type.

!

create_store_sales_order belum validasi stok saat insert — memang perilaku standar ERPNext, baru dicek saat Delivery Note.

!

Belum ada Customer Portal / self-service untuk partner store order sendiri dari luar. API saat ini untuk dipanggil sistem eksternal (POS/ordering app milik store), bukan diakses langsung end-user.

!

create_pos_sale belum menangani retur/void. Kalau transaksi dibatalkan di sisi store setelah ter-post, pembatalannya perlu ditangani manual (cancel Sales Invoice) atau butuh endpoint tambahan create_pos_return.
